Orbital Debris Collision Avoidance

AI / ML Aerospace Python TLE / SGP4
PROJ-001
Image 1 — System Architecture
System architecture diagram
Image 2 — Conjunction Plot
Predicted conjunction events (90-day window)
Problem Statement

Low Earth orbit is increasingly congested with defunct satellites and fragmentation debris. Current conjunction screening tools rely on manual operator review of CDMs (Conjunction Data Messages) and struggle to scale as catalog sizes grow beyond 27,000 tracked objects. False-positive rates exceed 95%, creating alert fatigue among operators.

The goal was to build an automated screening pipeline that reduces operator workload while maintaining zero missed high-probability events.

Solution & Implementation

Developed a Python pipeline ingesting live TLE data from Celestrak via the satellite.js SGP4 propagator, computing pairwise close-approach distances for all catalogued objects within a 90-day horizon. A gradient-boosted classifier (XGBoost) trained on historical CDMs filters the event stream, ranking events by predicted miss distance uncertainty.

Key components: automated TLE refresh every 6 hours, Cartesian conjunction geometry computation, feature engineering from orbital element deltas, and a Slack-integrated alert dispatcher. Deployed as a scheduled Python service on AWS Lambda.

Impact

Achieved a 78% reduction in false-positive conjunction alerts compared to the unfiltered CDM feed, with no missed events above 1-in-1000 collision probability threshold during a 60-day validation window.

0 Missed high-risk events

Structural Health Monitoring System

LabVIEW Automation Signal Processing DAQ
PROJ-002
Image 1 — LabVIEW Front Panel
Real-time LabVIEW front panel
Image 2 — FFT Fault Signature
FFT fault signature — bearing defect at 3× running speed
Problem Statement

Rotating machinery in aerospace ground support equipment degrades gradually through bearing wear, imbalance, and resonance buildup. Unscheduled maintenance from unexpected failures costs test campaigns weeks of delay and can corrupt test data through excessive vibration during sensitive measurements.

The facility needed a continuous, automated monitoring solution that could flag anomalies before failure and produce reports without human intervention.

Solution & Implementation

Built a LabVIEW application interfacing with an NI-9234 DAQ module sampling triaxial accelerometers at 51.2 kSa/s. The block diagram performs real-time FFT, RMS trending, and crest-factor calculation. A state machine monitors four health thresholds (ISO 10816 zones A–D) and triggers automated email alerts with attached spectrum PNG on zone C or D entry.

A report generation VI runs nightly, archiving trend data to a shared network drive in TDMS format and producing a PDF health summary. Configuration (sensor sensitivities, alarm thresholds, channel mapping) is stored in an XML file read at startup.

Impact

Deployed on three test rigs. Identified a developing bearing fault 11 days before failure in one rig, enabling a planned replacement during a scheduled maintenance window with zero test campaign disruption.

Autonomous Ground Rover

Arduino Robotics C++ PID Control
PROJ-003
Image 1 — Rover Hardware
Assembled rover with sensor array
Image 2 — Mapped Path
Serial-logged obstacle map — 4×4 m test arena
Problem Statement

Planetary surface exploration requires rovers capable of navigating unstructured terrain without constant ground-station commands. Latency makes teleoperation impractical beyond cislunar distances. This project investigated low-cost autonomous navigation algorithms that could later inform a larger-scale mission design study.

Solution & Implementation

Built around an Arduino Mega 2560 with three HC-SR04 ultrasonic sensors (front, left, right), dual-channel L298N motor driver, and an MPU-6050 IMU for heading estimation. Navigation logic runs a reactive obstacle avoidance loop: if the front sensor reads below 25 cm, the rover compares left/right clearance and turns toward the larger gap. A PID controller on the IMU yaw channel maintains heading during straight segments, reducing drift to under 3° per meter.

Obstacle positions are logged via Serial at 115200 baud and visualized in a companion Python script that plots a 2D occupancy grid in real time.

Impact

Completed 12 laps of a 4×4 m obstacle course autonomously with a 100% course completion rate. Heading drift reduced to 3° per meter vs. 18° without PID. Total component cost kept under $40.

Flight Anomaly Detection — Neural Network Classifier

AI / ML PyTorch Aerospace Time-Series
PROJ-004
Image 1 — Model Architecture
1D-CNN + LSTM encoder architecture
Image 2 — Confusion Matrix
Validation confusion matrix — 8 anomaly classes
Problem Statement

Flight data recorders generate hundreds of parameters at 64 Hz across thousands of flight hours. Manual review for anomaly investigation is prohibitively slow. Airlines and manufacturers need automated tools that can flag safety-relevant events — stall precursors, hydraulic pressure drops, engine surge signatures — without generating an overwhelming number of false alerts.

Solution & Implementation

Trained a hybrid 1D-CNN + LSTM model in PyTorch on synthetic FOQA (Flight Operational Quality Assurance) data covering 8 anomaly classes. Input windows of 512 time steps across 12 flight parameters are passed through three convolutional blocks (kernel size 7, 128 filters) then a bidirectional LSTM with 256 hidden units, followed by a softmax classification head.

Data augmentation (Gaussian noise injection, time warping) was applied to balance underrepresented classes. The model was evaluated with stratified 5-fold cross-validation. Inference runs at 2× real-time speed on a laptop CPU, enabling post-flight screening of full recordings.

Impact

Achieved 94.3% weighted F1-score across all 8 classes on held-out test data. The stall precursor class reached 97.1% recall — the highest-priority safety class. Processing a 10-hour flight record takes under 5 minutes end-to-end.

Automated Propulsion Test Bench

LabVIEW Automation Python DAQ
PROJ-005
Image 1 — Test Bench Setup
Thrust measurement bench with load cell array
Image 2 — Thrust Curve
Measured thrust curve vs. predicted — static fire #7
Problem Statement

Characterizing small solid and hybrid rocket motors requires precise simultaneous measurement of thrust, chamber pressure, temperature, and oxidizer mass flow. Manual data logging with separate instruments introduces synchronization errors and operator variability between test runs, making back-to-back comparative testing unreliable.

Solution & Implementation

Designed a LabVIEW test sequencer controlling an NI CompactDAQ chassis with modules for strain-gauge load cells (thrust), Type-K thermocouples (chamber + nozzle), pressure transducers, and a servo-driven pintle valve. All channels are sampled synchronously at 1 kSa/s with hardware-timed acquisition.

A Python post-processing script reads the TDMS output, applies load cell calibration curves, integrates thrust for total impulse, and generates standardized PDF reports and CSV exports. An interlock state machine in LabVIEW enforces a hardware abort if any sensor exceeds a safety threshold, venting oxidizer through a normally-open solenoid.

Impact

Used across 14 static fire tests with zero data synchronization errors. Measurement repeatability (1σ) improved from ±3.2% to ±0.4% compared to the prior manual logging approach. Setup time between test runs reduced from 45 minutes to under 10 minutes.
